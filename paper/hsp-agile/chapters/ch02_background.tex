\chapter{Background and Related Work}
\label{ch:background}

The pieces needed for prevention already exist---ordered FSF contracts,
LLM synthesis, CEGIS-style repair, SMT witnesses, pattern screens, mutation
metrics---but they have not been combined for specification-conformance
defects.
This chapter maps each piece briefly and ends with comparison tables that
position \textsc{SgDP}; formal objects wait for Chapter~\ref{ch:formalization}.

% ─────────────────────────────────────────────────────────────────────────────
\section{SOFL and Agile-SOFL}
\label{sec:bg:sofl}
% ─────────────────────────────────────────────────────────────────────────────

SOFL combines Z, data-flow diagrams, and OO structuring under stepwise
refinement~\citep{liu1998sofl,back1998refinement}; its guarded process
scenarios are the historical ancestor of FSF (Section~\ref{sec:bg:fsf}).
\emph{Agile-SOFL}~\citep{li2022qrs_companion} applies that rigour inside sprints:
FSF contracts, automatic conformance checking, and SMT test generation before
review~\citep{li2023iet_faultprevention,beck2001xp}.
HSP-Agile adds an LLM-driven synthesis--repair loop on top of that toolchain,
addressing the workflow-integration barrier noted by
\citet{woodcock2009formalsurvey}.

% ─────────────────────────────────────────────────────────────────────────────
\section{The Functional Specification Format (FSF)}
\label{sec:bg:fsf}
% ─────────────────────────────────────────────────────────────────────────────

FSF is the primary sprint artefact of Agile-SOFL~\citep{li2022qrs_companion,liu2004soflbook}.
A task names parameters, readable/writable externals, and an \emph{ordered}
sequence of scenario clauses---each a Boolean guard plus an assignment block;
the final clause may be an \texttt{others} catch-all and must appear last.
Guards are not a flat case statement: order is part of the contract.

\subsection{Scenario Ordering Semantics}
\label{sec:bg:fsf:semantics}

The defining rule is \emph{first-match}: the first satisfied guard determines
the output.
On \code{classify\_signal} (Chapter~\ref{ch:intro}), input
$(\mathit{level},\mathit{threshold})=(-3,-10)$ satisfies both $S_1$
($\mathit{level}<0$) and $S_2$ ($\mathit{level}\ge\mathit{threshold}$), yet
FSF requires \texttt{"Error"} because $S_1$ has priority.
An implementation that tests the threshold first returns
\texttt{"Critical"}---a \emph{scenario-order violation} invisible to checkers
that treat guards as an unordered Boolean formula.
Chapter~\ref{ch:formalization} defines the oracle $g_t$ and the precedence
partition formally (Figure~\ref{fig:fsf-semantics}); unlike Dijkstra's guarded
commands~\citep{dijkstra1975guarded}, FSF is deterministic priority, not
nondeterministic choice.

\section{LLM-Assisted Code Generation}
\label{sec:bg:llm}
% ─────────────────────────────────────────────────────────────────────────────

Contest-style generation optimises for pass@k on hidden
tests~\citep{chen2021codex,li2022alphacode,openai2023gpt4}.
FSF synthesis needs something stricter: guaranteed conformance to an ordered
scenario contract.
Deployed assistants often produce \emph{semantically plausible} code that
fails at boundaries and multi-condition
logic~\citep{dakhel2023copilot}---exactly what FSF guards encode.

Without additional guidance, three failure modes recur
\citep{dakhel2023copilot,sun2024clover}:
(i)~\emph{scenario-order blindness}---overlapping guards bias the model toward
``natural'' \texttt{if} order rather than mandated precedence;
(ii)~\emph{boundary hallucination}---confused relational operators at
scenario edges (e.g.\ $<$ vs.\ $\le$);
(iii)~\emph{spurious completeness}---hard-coded defaults that pass superficial
tests (pattern RF07; Section~\ref{sec:bg:patterns}).
Test-feedback repair~\citep{chen2021codex} helps only when tests hit the
violated region; fixed suites often miss overlaps.
HSP-Agile therefore synthesises boundary witnesses with Z3 rather than relying
on sampled unit tests alone.

% ─────────────────────────────────────────────────────────────────────────────
\section{Counterexample-Guided Synthesis (CEGIS)}
\label{sec:bg:cegis}
% ─────────────────────────────────────────────────────────────────────────────

Repair needs counterexamples that name the violated region, not only a failing
assertion.
CEGIS splits synthesis into a synthesiser and a verifier that returns
counterexamples until success or budget exhaustion
\citep{solarlezama2008sketching}.
LLM-augmented variants replace templates with free-form generation while
retaining formal checks~\citep{loughman2024dafnybench,sun2024clover},
but prior systems lack FSF first-match semantics and pattern-guard pre-filters.

HSP-Agile instantiates the \textsc{SgDP} loop with an LLM synthesiser, a
Z3-based verifier over suite $\mathcal{D}(t)$, and a feedback aggregator that
packages counterexamples $(x^*, g_t(x^*), f_{c_k}(x^*))$ together with
high-severity pattern hits (Figure~\ref{fig:cegis-loop}).
Budget $K$ caps iterations ($K=3$ in the experiments); on exhaustion the
highest-scoring candidate is returned as best-effort~\citep{barrett2018smt}.

% ─────────────────────────────────────────────────────────────────────────────
\section{SMT Solving and Formal Verification}
\label{sec:bg:smt}
% ─────────────────────────────────────────────────────────────────────────────

SMT extends SAT with background theories; a model is a concrete test
input~\citep{barrett2018smt,demoura2008z3}.
For each scenario $s_i$, HSP-Agile solves the \emph{effective} guard
$\varphi_i = g_i(x) \wedge \bigwedge_{j<i}\neg g_j(x)$, guaranteeing at least
one witness per first-match region---targeted coverage that random testing
(e.g.\ QuickCheck~\citep{claessen2000quickcheck}) does not provide.
This is weaker than full deductive verification (e.g.\ Dafny~\citep{leino2010dafny})
but requires no developer annotations and fits Agile-SOFL sprint velocity;
prevention guarantees remain bounded by suite coverage
(Chapter~\ref{ch:formalization}).

% ─────────────────────────────────────────────────────────────────────────────
\section{Requirement-Level Error Pattern Detection}
\label{sec:bg:patterns}
% ─────────────────────────────────────────────────────────────────────────────

Static analysers encode recurring structural bug
patterns~\citep{hovemeyer2004findbugs,cousot1977abstract}.
HSP-Agile's pattern guard lifts the idea to \emph{requirements-level}
violations: signatures that cannot conform to FSF regardless of tested inputs.
Catalogue RF01--RF14~\citep{li2023iet_faultprevention} (Appendix~\ref{app:patterns}) includes
missing preconditions, wrong defaults, swapped comparisons, and unconditional
success (RF07); high-severity hits ($\mathcal{P}^H$) block acceptance and enter
CEGIS repair context even when all formal tests pass.

% ─────────────────────────────────────────────────────────────────────────────
\section{Mutation Testing for Specification Assessment}
\label{sec:bg:mutation}
% ─────────────────────────────────────────────────────────────────────────────

Mutation testing injects syntactic faults; survivors indicate inadequate
coverage~\citep{demillo1978mutation,offutt2001mutation2000,ammann2016testing}.
HSP-Agile uses two operator families (details in Chapter~\ref{ch:experiments}):
specification-level SNO, ORO, SPO; implementation-level ICO, WRO, BCO.
Prevention quality is scored as \emph{Prevention Detection Rate} (PDR: fraction
of injected faults correctly rejected) and \emph{False Accept Rate} (FAR:
fraction of conforming candidates incorrectly rejected), analogous to
sensitivity--specificity~\citep{okun2013sate4}.

\section{Related Work}
\label{sec:bg:related}
% ─────────────────────────────────────────────────────────────────────────────

What is missing is not another synthesiser, but a prevention loop that treats
ordered guards as first-class evidence.
Table~\ref{tab:related-work} is the short answer.

Successful formal-method deployments automate narrow, high-value layers inside
existing workflows~\citep{woodcock2009formalsurvey,pohl2010requirements};
Agile-SOFL follows that prescription, and HSP-Agile automates synthesis and
repair without manual annotation.

Closest LLM+verifier systems differ on three axes: whether witnesses respect
first-match order, whether repair carries scenario structure, and whether
prevention (not only pass@k) is measured.
DafnyBench~\citep{loughman2024dafnybench} uses full deductive verification but
repairs from compiler messages, with no pattern guard and pass@k evaluation.
Sun et al.\ (Clover)~\citep{sun2024clover} is the closest verifier-in-the-loop
design, yet lacks FSF ordering, pattern guards, and PDR/FAR.
Baselines B1 (one-shot LLM) and B2 (test-feedback without SMT witnesses)
isolate those two ingredients.

\section{Gap Summary}
\label{sec:bg:gap}
% ─────────────────────────────────────────────────────────────────────────────

LLM suggestions still harbour subtle semantic defects that coarse suites miss
\citep{chen2021codex,openai2023gpt4,dakhel2023copilot}.
Prior SOFL prevention work assumed human-written
implementations~\citep{li2022qrs_companion,li2023iet_faultprevention}; LLM synthesis creates failure
modes neither human-process prevention nor test-only post-processing alone
addresses.
Coding agents (OpenHands, SWE-agent, and similar)~\citep{wang2024opendevin,yang2024sweagent}
target repository tasks without a specification oracle $g_t$ or PDR/FAR;
they are orthogonal to \textsc{SgDP} (Reflexion-style verbal memory captures a
dimension most relevant to repair but is not evaluated here).

Table~\ref{tab:related-positioning} positions \textsc{SgDP} on the dimensions
that matter for specification-conformance defect prevention.

Unlike DafnyBench-style proof pipelines~\citep{loughman2024dafnybench}, the
model is not asked for annotations it systematically fails to produce.
Unlike Self-Refine / Self-Debug verbal loops, every repair step is grounded in
an executable oracle, and evaluation measures prevention (PDR/FAR), not only
generation success.

The remaining gap is therefore not a longer feature list.
It is three insights that Table~\ref{tab:related-positioning} leaves open and
that Chapter~\ref{ch:formalization} makes precise:
\begin{enumerate}
  \item[\textbf{C1.}] Prevention must depend on \emph{semantic regions}
        (first-match partitions), not surface syntax---realised by SpecIR.
  \item[\textbf{C2.}] Repair needs a \emph{typed scenario failure}, not raw
        pass/fail bits---realised by the Semantic Feedback IR.
  \item[\textbf{C3.}] Release must be \emph{conjunctive}: every witness matches
        \emph{and} no high-severity structural pattern remains.
\end{enumerate}
