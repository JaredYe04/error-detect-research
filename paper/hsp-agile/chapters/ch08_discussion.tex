\section{Interpretation of Core Findings}
\label{sec:disc:interpretation}
%=============================================================================

\paragraph{Practical takeaway.}
\textbf{Tier-1 for practitioners:} Table~\ref{tab:deployment-boundary} is the
primary deliverable---match feedback depth to specification structure.
Mechanistically: E6 (+7.7\,pp typed feedback) and E4 (SMT boundary coverage)
show where specification structure helps; conjunctive acceptance trades
strict-success rate for lower escaped defects.
On aggregate conformance and unfiltered corpora (E10, E12), test-feedback B2 is
the default; full mode~M is warranted when conjunctive release or prevention
(PDR/FAR) is required---not from overlap tertiles alone (canonical E3 is
non-monotone).

The central stress-test result is that \textbf{typed feedback} (E6) and
\textbf{overlap-conditional} witness repair (E3, Table~\ref{tab:e3-tiers}) address
ordered-guard defects tests miss.
E1 canonical (seed~0): B1 89.2\%, B2 87.7\%, M 86.3\%---no mode beats another
at Holm $p{<}0.05$.
E12-full favours B2 on all three seeds (Section~\ref{sec:results:e12}); single-repeat
E1 is not a ranking claim.
Mapped to insights: E3/E4 bound C1 (overlap tertiles are non-monotone on
canonical E3; E4 shows SMT maintains higher boundary coverage than random
sampling); E6 supports C2 (+7.7\,pp for full Semantic Feedback IR);
E1 Strict / E2 PDR--FAR support C3 (acceptance stays selective while
prevention improves).

B0 one-shot (89.2\% mean Conf., $\approx$0.2\,s) matches B1 at seed~0 on
canonical E1---a one-call reference, not a production baseline.
Ablations: A3 $-$6.3\,pp (no repair), A1 $-$3.9\,pp (no checker), A2
$-$6.0\,pp on Conf.\ (pattern guard is a C3 prevention/diagnostic gate, not a
Conf.\ booster; Section~\ref{sec:disc:implications}).

%-----------------------------------------------------------------------------
\subsection{Named Failure Patterns}
\label{sec:disc:failure-patterns}
%-----------------------------------------------------------------------------

Five recurring failure patterns account for the bulk of the 90 residual errors.

\paragraph{Arithmetic reasoning failure.}
The LLM produces a formula that is structurally correct (correct branches,
correct variable references) but arithmetically wrong---for example, computing a
weighted sum with the wrong denominator or using integer division where float
division is required.
HSP-Agile misses these because Z3 witnesses are generated from guard predicates,
not from output expressions; the formal checker therefore cannot witness an
expression-level arithmetic fault unless it happens to coincide with a guard
boundary.

\paragraph{Long-range dependency.}
Some FSF specifications require the output of scenario~$i$ to depend on the
\emph{cumulative} effect of scenarios $1, \ldots, i-1$ (e.g.\ a running total or
a sequence-sensitive state machine transition).
The repair loop issues one witness per scenario in isolation; it cannot construct
a multi-step witness that would expose the accumulated-state fault.
This structural limitation of the E9 formal checker (single-scenario witnesses)
is the primary driver of \emph{OrderingError} and \emph{StateError} failures.

\paragraph{State update confusion.}
When two or more FSF scenarios modify the same output variable, the LLM
occasionally generates code in which later scenarios overwrite rather than
accumulate the earlier value.
The counterexample-guided repair prompt identifies the violated scenario but does
not explicitly communicate the accumulation invariant, so the LLM corrects the
reported case while re-introducing the error in an adjacent scenario.
Extending the Semantic Feedback IR with an explicit \texttt{accumulation\_mode}
field would address this pattern.

\paragraph{Unsatisfiable guard (Unsat guard).}
A small number of FSF specifications contain scenario guards that Z3 proves
mutually satisfiable but that the LLM interprets as mutually exclusive---producing
dead branches or guard conditions that can never be simultaneously true.
Because the implementation is syntactically valid and passes individual-scenario
witnesses, the formal checker does not flag the dead branch; only a path-coverage
oracle (not currently part of the pipeline) would detect it.

\paragraph{Hallucinated API call.}
The LLM occasionally calls a library function that does not exist in the
permitted API surface defined by the FSF signature---for example, calling
\texttt{math.log\_base(x, b)} instead of \texttt{math.log(x) / math.log(b)}.
Pattern guard RF07 catches this post-generation, but the repair loop does not
receive a structured signal about the illegal call; it receives only a generic
runtime error counterexample, leading to repeated hallucinations across repair
attempts.
A dedicated RF07-triggered repair template that injects the legal API surface into
the repair prompt would break this cycle.

\section{The Quality--Pass-Rate Paradox}
\label{sec:disc:paradox}
%=============================================================================

One of the most important results on current BENCH-120 is that \emph{strict
success is uniformly low} across LLM repair modes (M/B2/B1 at 5.0/5.0/4.2\%)
while \emph{mean conformance} still discriminates B2 $>$ M $>$ B1.
The two metrics measure fundamentally different things.
\emph{Strict success rate} is binary per task: conjunctive
$\mathrm{Accept}$ requires full witness conformance \emph{and} a clear pattern
screen.
\emph{Formal conformance} is continuous: a candidate satisfying 9 of 10 cases
contributes 0.9 to the average even though it contributes 0 to strict success.

On the current corpus, B1/B2 lead aggregate conformance at seed~0 (89.2\%/87.7\%);
M reaches 86.3\%; the historical ``higher Conf.\ but lower Strict'' paradox applied only
to the superseded 86-task-overlapping E1 snapshot (M 90.4\% vs.\ B2 88.0\% with
25.0\% vs.\ 26.7\% Strict).
The enduring lesson is C3 selectivity: acceptance stays rare under conjunctive
release, and prevention metrics (PDR/FAR, E2)---not strict-success rate
alone---should guide deployment.

A useful analogy clarifies this relationship.
Suppose two students sit ten examinations.
Student~A scores 89\% on every paper.
Student~B scores 100\% on three papers and 65\% on the remaining seven.
On a binary pass/fail threshold of 75\%, Student~A passes all ten and
Student~B passes only three.
Yet Student~B's \emph{mean score} is $0.3 \times 100 + 0.7 \times 65 = 75.5\%$,
below Student~A's 89\%.
The binary metric favours Student~B despite lower average quality; the continuous
metric correctly orders them.
M is Student~A: consistently high conformance; B1 and B2 occasionally solve tasks
completely but average lower quality.

This is not a deficiency of M.
The conjunctive predicate rejects candidates that pass all witnesses but exhibit
high-severity pattern violations---candidates that B1/B2's single-criterion gate
would accept.
The escaped-defect rate confirms that M's accepted set contains 3.8\,pp fewer
mutant-detectable faults than B2's accepted set (FAR 5.0\% vs.\ 8.8\% on
implementation screening), while achieving higher detection rate (PDR 95.0\% vs.\
91.2\%).
The Wilcoxon signed-rank test on strict success rate yields uncorrected
$p = 0.921$ (Holm-corrected $p = 1.000$; not significant), justifying formal
conformance as the \emph{primary} metric on the hard benchmark.

\section{Implications for Practice}
\label{sec:disc:implications}
%=============================================================================

\subsection{Sprint Integration and CI/CD Deployment}

The \textsc{SgDP} framework is designed as a CI/CD pre-merge gate operating between
test-feedback (fast, spec-unaware) and full deductive verification (slow, requires
annotations).
Mode~M incurs mean latency of 29{,}378\,ms per task ($3.2\times$ B2's 9{,}082\,ms),
reflecting repair-loop LLM calls and Z3 checking.
For \emph{interactive} IDE use, 29 seconds is at the upper boundary of acceptable
response times; for \emph{CI/CD pre-merge gates}, it is well within typical automated
quality checks.
A sprint team merging 20--30 LLM-assisted units per day would incur roughly
10--15 minutes of additional pipeline time---comparable to compiling a medium-sized
Java project---while avoiding hours of downstream debugging from propagated
conformance faults.

HSP-Agile is best deployed as a background pre-merge quality gate rather than
an interactive code-completion tool, consistent with agile fitness-function
practice~\citep{harman2004maff,beck2001xp}.
The conjunctive acceptance predicate produces an auditable decision record---not
just pass/fail---enabling developers to inspect which witnesses and patterns were
decisive.

\subsection{The Formal Checker as a Lightweight Oracle}

The Z3-based conformance checker occupies a middle ground between no formal
verification and full deductive verification.
Full tools such as Dafny~\citep{leino2010dafny} and Coq provide complete guarantees
but require annotation-heavy specifications and trained verification
engineers~\citep{woodcock2009formalsurvey}.
The Z3 checker provides \emph{bounded} guarantees over a finite generated test
suite at less than 100\,ms per task---negligible relative to total pipeline latency.
This lightweight oracle suits iterative formal development workflows: test cases are derived
directly from specification structure, not manually written unit tests.
The $+3.9$\,pp conformance contribution of the checker (A1 ablation) demonstrates
material quality improvement, not bureaucratic overhead.

\subsection{Pattern Guard Selectivity and Advisory Use}

The A2 ablation---removing the pattern guard costs $\approx$6\,pp mean conformance on
canonical E1---confirms the guard is not a Conf.\ booster.
The guard provides value as a \emph{classification and diagnostic tool} and
prevention lever rather than a mean-Conf.\ driver: when it fires, it identifies the specific fault pattern
(e.g.\ RF07 unconditional success; RF03 missing precedence guard), information not
captured by the conformance metric.

For most patterns, \emph{advisory warnings} are preferable to hard rejections,
following static-analysis severity-modulated gating
policy~\citep{hovemeyer2004findbugs,ayewah2007findbugs}, with hard rejection reserved
for RF07.
The pattern guard is most valuable as a \emph{living artefact}: new failure modes
observed in production can be added incrementally as RF patterns without retraining
the LLM or modifying the formal checker; E7's per-pattern F1 guides catalogue
expansion.

\section{Generalisation Boundaries}
\label{sec:disc:generalisation}
%=============================================================================

The framework requires deterministic, enumerable guard conditions encodable as SMT
constraints, covering FSF-style ordered scenarios, finite state machine transition
guards, and simplified Z/TLA+ preconditions over bounded domains.
Generalisation experiments (E8) demonstrate that M consistently outperforms B1
across SOFL/FSF, Mini-StateMachine, and Mini-Z notations with only adapter-level
changes, supporting the framework claim rather than a single-notation-specific claim.
On the expanded $n{=}30$ adapter corpora (\texttt{run\_e8b\_expanded\_v1}), absolute
conformance is low (23--42\%) because Mini-Z and Mini-StateMachine syntax impose
generation constraints beyond FSF overlap structure; nevertheless M leads B1 on both
notations and matches or exceeds B2 (Mini-StateMachine: 41.6\% vs.\ 40.8\%;
Mini-Z: 23.3\% vs.\ 18.3\%).
Real-derived HumanEval/MBPP tasks (E8c) show stronger transfer: M reaches 95.0\%
on MBPP-FSF (tied with B2) while B2 leads on the sparser HumanEval-FSF subset
(98.8\% vs.\ 87.1\%).
These boundary cases do not undermine the mechanism result (E6 +7.7\,pp typed
feedback) or E12 ranking stability (B2 leads all seeds) but clarify that
framework components should be enabled selectively based on specification
complexity and notation familiarity.

\paragraph{When does B2 (test-feedback) outperform M?}
B2 is more effective than the full HSP-Agile pipeline under three typical conditions.
First, guard regions are sparse or exhibit little overlap, allowing randomly generated test cases to cover decision boundaries with high probability, as observed on HumanEval-FSF (E8c) and the E11 external corpus.
Second, the number of branches is limited (typically no more than four), which reduces the likelihood of incorrect guard ordering by the LLM, consistent with the real-derived HumanEval and MBPP tasks.
Third, output expressions consist of simple scalar values rather than coupled tuples, making branch assignment errors readily detectable through test failures.
Under these conditions, the lightweight feedback mechanism of B2 achieves
conformance comparable to, or exceeding, that of M while reducing inference
latency by $3.3\times$ on canonical E1 (13.9\,s vs.\ 46.2\,s).

M's value on the hard benchmark is therefore not universal aggregate dominance
but \emph{conditional}: E6 (+7.7\,pp typed feedback under controlled comparison),
E2 (PDR/FAR), and conjunctive $\mathrm{Accept}$ when those release bars apply.
Overlap tertiles on canonical E3 are non-monotone; E10 aggregate favours B2.
These observations motivate the deployment table rather than a single overlap
threshold rule.

\section{Limitations}
\label{sec:disc:limitations}
%=============================================================================

Several limitations bound the scope of the claims made in this report.

\paragraph{Benchmark representativeness.}
The hard benchmark consists of 120 synthetic formal specification tasks (SOFL/FSF
instantiation) from a fixed template family designed to probe precedence-sensitive
scenario orderings.
Industrial specifications may exhibit more complex state dependencies, external
service interactions, or composite scenario structures not represented in the
benchmark.
Replication on practitioner SOFL/FSF projects~\citep{li2022qrs_companion,pohl2010requirements}
is necessary before broad generalisability can be claimed.

\paragraph{Single LLM family.}
All experiments used Qwen2.5-27B-Instruct via an OpenAI-compatible LLM gateway.
Conformance gains may differ with GPT-4, Claude~3.5 Sonnet, or Gemini~1.5 Pro.
The pipeline structure is model-agnostic, but quantitative results are specific to
the evaluated model family.

\paragraph{Z3 test-case coverage.}
The formal checker generates one test case per scenario branch.
Denser witness generation (boundary-value analysis, perturbation around Z3 witnesses)
would increase fault detection sensitivity at higher latency.
The current regime likely underestimates M's conformance advantage relative to a
denser test suite.

\paragraph{Prevention evaluation completeness.}
PDR/FAR analysis in Chapter~\ref{ch:results} is based on the full prevention
evaluation (\code{prevention\_full\_v1}, 1{,}704 mutation records per mode).
M achieves PDR 95.0\% and FAR 5.0\% on implementation screening, compared to
B2's PDR 91.2\% and FAR 8.8\%, confirming that the conjunctive acceptance gate
reduces escaped defects without sacrificing detection sensitivity.

\paragraph{Repair loop convergence.}
The maximum repair budget $K$ is fixed across all tasks.
Harder tasks may benefit from larger $K$; simpler tasks waste budget on unnecessary
iterations.
Adaptive budget allocation based on observed conformance improvement per iteration
could reduce latency while maintaining quality on the hardest tasks.

\section{Comparison to Related Approaches}
\label{sec:disc:related}
%=============================================================================

\paragraph{DafnyBench and full deductive verification.}
DafnyBench~\citep{loughman2024dafnybench} evaluates LLM synthesis of Dafny programs
with machine-checked correctness proofs.
Full deductive verification provides stronger guarantees than HSP-Agile's bounded Z3
checking but requires formal annotations absent from FSF specifications.
HSP-Agile is applicable directly to SOFL/FSF inputs without verification
annotations, making it usable by practitioners not trained in deductive verification.
The two approaches occupy complementary positions in the verification-cost spectrum.

\paragraph{CEGIS-inspired approaches for Dafny (Sun et al., Clover).}
Sun et al.\ (Clover)~\citep{sun2024clover} apply counterexample-guided inductive
synthesis (CEGIS) to assist LLMs in producing correct Dafny programs.
Both approaches use witnessed failures to construct targeted repair prompts; the key
difference is that Sun et al.\ (Clover) operate in the Dafny verification context with
proof-obligation counterexamples, whereas HSP-Agile generates counterexamples from
Z3-driven test cases derived from FSF scenario semantics.
HSP-Agile's checker is lighter-weight but provides weaker guarantees as a consequence.

\paragraph{VerifierLoop-style FSF baseline (B6).}
Mode~B6 (VerifierLoop-FSF) implements the closest related-work comparator we
deferred in the pre-revision draft: SMT witness checking inside the repair loop
with structured counterexample prompts, equal $K{=}3$ and witness budget as M,
but without Semantic Feedback IR or pattern guard
(\texttt{src/baselines/verifier\_loop\_fsf.py}).
On the 30-task stratified subset, B6 matches B2 (86.0\% vs.\ 86.2\% Conf) while
M trails (83.3\%)---aggregate ranking on small $n$ is seed-sensitive (cf.\ E12).
On the full 100-task hard export, B6 reaches 78.6\% Conf versus M at 86.3\% on
E1's 120-task matrix, isolating the gain from specification-indexed IR fields
beyond raw witnesses.
The originally cited ``VerifierLoop'' TOSEM entry was removed after bib
verification; Sun et al.'s Clover~\citep{sun2024clover} remains the verified
CEGIS reference for Dafny, not a drop-in FSF comparator---B6 is our FSF adaptation.

\paragraph{Standard LLM baselines (B1 and B2).}
Full mode~M trails B1/B2 on canonical E1 aggregate conformance ($-$2.9\,pp vs.\
B1; $-$1.5\,pp vs.\ B2; both Holm $p{=}1.0$) at approximately $3.3\times$
latency over B2~\citep{dakhel2023copilot,chen2021codex}.
Per-task win rate and E12 multi-seed stability should guide adoption, not the
headline mean alone.
HSP-Agile occupies a ``high quality, moderate cost'' regime: appropriate for quality
gates in workflows where downstream formal verification costs dominate, not for
real-time code-completion.

\section{Threats to Validity}
\label{sec:disc:threats}
%=============================================================================

Threats-to-validity analysis follows the four-category framework
common in empirical software engineering~\citep{ammann2016testing,woodcock2009formalsurvey}.

\paragraph{Construct validity.}
Formal conformance is approximated via a finite witness set $\mathcal{D}(t)$;
candidates outside the checked space are not guaranteed correct.
\emph{Mitigation:} Theorem~\ref{thm:far-bound} quantifies the FAR bound, and E7
reports per-pattern P/R/F1 so catalogue gaps are explicit.
PDR uses first-order mutants; higher-order mutations may differ.

\paragraph{Internal validity.}
LLM non-determinism and author-built benchmarks are the main risks.
\emph{Mitigation:} $T_\text{repair}=0.0$, fixed seeds, bootstrap CIs over
tasks, E12-full multi-seed stability (120 tasks $\times$ 3 seeds),
the Z3 satisfiability filter,
E10 (no overlap filter), the real-derived E8 subset, extended B3--B5 baselines,
and ablations A1--A3.
E3 complexity stratification checks that the SMT advantage is not an artefact of
a single difficulty band.

\paragraph{External validity.}
Primary evaluation uses 120 synthetic numeric-domain Python tasks and one LLM.
\emph{Mitigation:} E8 exercises Mini-Z and Mini-StateMachine via adapters;
E10 samples uniformly without the overlap filter; E8c reports full B1/B2/M on
HumanEval/MBPP-FSF (\texttt{run\_e8c\_full\_v1}).
Language- or model-universal results are not claimed beyond that evidence.

\paragraph{Conclusion validity.}
All pairwise comparisons use Holm-Bonferroni correction.
\emph{Mitigation:} the non-significant strict-success comparison
($p \approx 0.92$) is treated as expected under the conjunctive gate, with
conformance as the primary metric; full E2 supersedes preliminary $n=40$
prevention numbers.
Post-hoc power analysis (Section~\ref{sec:exp:stats}) shows $n{=}120$ is
under-powered for M vs.\ B1 ($\delta{=}{-}0.036$, observed power ${\approx}5.8\%$)
and for M vs.\ B2 ($\delta{=}{-}0.014$, observed power ${\approx}3.6\%$).

\subsection{Anticipated Reviewer Questions}
\label{sec:disc:threats-faq}

The following questions are answered explicitly so threats are not buried in
appendix-only material.

\paragraph{Why $n{=}120$? Why not $n{=}240$?}
The hard benchmark balances overlap-filtered discriminative tasks with LLM API
cost ($\geq$1{,}200 E1 jobs at repeat~0); B0 one-shot (89.2\% mean Conf.) anchors
the one-call reference without repair overhead.
Post-hoc power for M vs.\ B2 at the observed effect size is ${\approx}3.6\%$
(\texttt{power\_analysis.json}); detecting $|\delta|{=}0.014$ would require
$n{\gtrsim}39{,}000$ paired tasks for 80\% power---doubling to $n{=}240$ would
not materially change inference.
We therefore supplement aggregate means with per-task win rate (4/120
mean-across-seeds), E10
(uniform random sample), E12-full (120-task multi-seed), and B6
(VerifierLoop-FSF).

\paragraph{Why the Z3 overlap filter?}
The filter retains tasks where first-match ordering defects are \emph{possible}:
without overlapping guards, precedence bugs are rare and test-feedback baselines
are advantaged.
E10 removes the filter to bound this bias; E3 stratifies by overlap tertile on
the hard set.
Claims are scoped to overlap-intensive specifications, not universal codegen.

\paragraph{Why is strict success only 5.0\%?}
$\mathrm{Accept}$ is conjunctive (Equation~\eqref{eq:accept}): full witness
conformance \emph{and} a clear high-severity pattern screen.
On current BENCH-120, all LLM modes share similarly low strict rates
(M/B2/B1 at 5.0/5.0/4.2\%)---C3 selectivity, not weaker implementations.
Prevention metrics (PDR/FAR, E2) differentiate M from B2 when conjunctive
release is required (Section~\ref{sec:disc:paradox}).

\paragraph{Why keep the pattern guard when A2 is faster and slightly higher on Conf.?}
A2 reaches 80.3\% Conf.\ vs.\ M's 86.3\% at $\approx$1.2$\times$ lower latency
because it skips hard rejections on structural patterns.
The guard's value is prevention (PDR/FAR, E2), not mean Conf.\ (A2 ablation
$+$0.5\,pp on Conf.\ but lower structural PDR).
For release gates, advisory RF patterns plus hard rejection for RF07 are the
intended operating mode (Section~\ref{sec:disc:implications}).

\paragraph{Why a single seed for E1?}
Generative calls use $T_\text{gen}{=}0.7$; repair uses $T_\text{repair}{=}0.0$.
E1 reports repeat~0 with bootstrap CIs over tasks; \textbf{E12-full} re-runs all
120 tasks across three seeds for B1, B2, and M (Section~\ref{sec:results:e12}).
B2 leads in every seed; M wins only 4/120 paired comparisons (Friedman
$p{=}0.077$; ranking stable).
The single-repeat matrix is not a one-seed artefact, but pooled seeds favour B2
over M---deployment should follow overlap tier, not seed~0 alone.

\paragraph{Why does A2 beat M on mean conformance?}
A2 disables the pattern guard, so candidates that pass all witnesses but trigger
a high-severity pattern are still counted toward mean Conf.\ even when M would
withhold $\mathrm{Accept}$.
A2 is the right mode when latency binds and prevention gating is deferred; M is
the right mode when conjunctive release is required.

\section{SpecIR Scope Assumptions and Future Extensions}
\label{sec:disc:specir-scope}

The current SpecIR design (Definitions~\ref{def:specir}--\ref{def:specir-wf})
makes explicit assumptions that bound framework generality.
Table~\ref{tab:specir-scope} maps each assumption to a natural extension;
Section~\ref{sec:conc:future} prioritises these directions for future work.

\begin{table}[t]
  \centering
  \caption{\textbf{SpecIR scope assumptions and extension paths.}
    Each row states a current design choice, its limitation, and a research
    direction beyond the HSP-Agile evaluation.}
  \label{tab:specir-scope}
  \footnotesize
  \renewcommand{\arraystretch}{1.15}
  \begin{tabular}{@{}p{0.22\linewidth}p{0.30\linewidth}p{0.38\linewidth}@{}}
    \toprule
    \textbf{Assumption} & \textbf{Limitation} & \textbf{Extension} \\
    \midrule
    Deterministic guards &
    Cannot express probabilistic thresholds or noisy sensors &
    Probabilistic guards with confidence intervals \\
    First-match ordering &
    No temporal ``always/eventually'' constraints &
    LTL/CTL adapters over transition systems \\
    Single oracle $g_t$ &
    No distributed or multi-service contracts &
    Compositional oracles with interface contracts \\
    Linear integer arithmetic (LIA) &
    Non-linear and floating-point guards fail encoding &
    NIA/QF\_FP backends with staged fallback \\
    \bottomrule
  \end{tabular}
\end{table}

These assumptions are deliberate scoping choices for a first framework
instantiation, not fundamental barriers.
E8 generalisation across Mini-Z and Mini-StateMachine shows that adapter-level
extensions preserve the shared pipeline; temporal and probabilistic semantics
would need new adapter and witness modules while retaining the Semantic
Feedback IR and acceptance predicate.
What has been established, and what remains open for richer encodings and
industrial tasks, is summarised next.
