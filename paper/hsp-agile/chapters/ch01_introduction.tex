\chapter{Introduction}
\label{ch:intro}

Formal specifications give requirements a precision that informal prose cannot
match.
When large language models (LLMs) implement those specifications, that precision
becomes a liability: the model has no native notion of guard order or
first-match semantics, so it produces code that is statistically plausible yet
formally wrong.
We call the resulting failures \emph{specification-conformance defects}---faults
that pass most everyday inputs but break exactly where the specification draws
its boundaries.
Test-based repair loops miss these defects because they sample the input space
without knowing which regions the specification treats as critical.

This report asks: \emph{can we prevent specification-conformance defects in
LLM-generated code by using the specification itself as the primary evidence
source, without requiring full deductive verification?}
We answer yes.
We present the \textbf{Specification-guided Defect Prevention Framework}
(\textsc{SgDP}), with \textbf{HSP-Agile} as its primary instantiation over
ordered guard-based specifications (SOFL/FSF in our benchmark).


\section{Problem Statement}
\label{sec:intro:problem}

\subsection{Setting}
The Structured Object-oriented Formal Language (SOFL)
\cite{liu1997sofl,liu2014sofl} provides the Functional Specification Format
(FSF): an ordered list of scenarios, each pairing a guard predicate with a
postcondition under first-match precedence.
Agile-SOFL \citep{li2022agilesofl} embeds FSF tasks in sprint workflows as
design documents, oracles, and acceptance criteria.
LLM-assisted generation accelerates that workflow
\cite{chen2021codex,openai2023gpt4}, but synthesises by pattern completion
rather than by respecting guard order
\cite{dakhel2023copilot,lynn2024verifierloop}.
Overlapping guards therefore expose precedence-sensitive faults that standard
test suites miss \citep{zhu1997specification}.

\subsection{What Goes Wrong}
In plain terms, a specification-conformance defect is a bug that unit tests
rarely catch.
The generated function may look correct on the ``happy path,'' yet on a thin
slice of the input space---often where two guards overlap---it returns the
output of the wrong scenario.
Random or distribution-biased tests almost never land in that slice, so a
repair loop that only sees failing unit tests never learns about the ordering
constraint.

Formally, the release decision is a bounded acceptance predicate.
Given task~$t$ with ordered scenarios $\mathcal{S}_t = (S_1,\ldots,S_n)$ and
synthesis budget~$K$, we ask whether any candidate~$c$ satisfies
\begin{equation}
  \mathrm{Accept}(c, t)
  \;\iff\;
  \mathrm{Conf}(c, t) \geq \tau
  \;\wedge\;
  P^H(c) = 0,
  \label{eq:acceptance}
\end{equation}
where $\mathrm{Conf}(c,t)\in[0,1]$ is the scenario-level conformance score,
$\tau=1.0$ in this work, and $P^H(c)=0$ asserts absence of high-severity fault
patterns under static screening.
Building a witness for scenario~$S_i$ requires an input that satisfies
$G_i(\mathbf{x}) \wedge \bigwedge_{j<i}\neg G_j(\mathbf{x})$---a conjunction
that standard sampling heuristics cannot reliably reach
\cite{barrett2018smt}.

Consider the \code{classify\_signal} specification from our benchmark
(Listing~\ref{lst:fsf-spec}).
We reuse this task as the running example throughout the report.

\section{Research Challenges}
\label{sec:intro:challenges}

Three challenges stand between the example above and a usable prevention
pipeline.

\subsection{Closing the Specification--Code Semantic Gap}
Verifying conformance to scenario~$S_i$ requires inputs in
$G_i \wedge \bigwedge_{j<i}\neg G_j$, often a low-measure region missed by
random sampling.
SMT solvers \citep{demoura2008z3,barrett2018smt} can generate such witnesses,
but only if the FSF encoding is faithful.
\textsc{SgDP} addresses this with a \emph{constraint-guided witness generator}
that encodes guard precedence as SMT constraints
(Chapter~\ref{ch:method}).

\subsection{Structured Counterexample-Guided Repair}
Test-feedback loops lack specification-level structure
\cite{lynn2024verifierloop}: the LLM may patch one counterexample without
fixing guard precedence.
Effective repair needs prompts that name the violated scenario, state the
ordering constraint, and include the SMT witness.
\textsc{SgDP} addresses this with a \emph{scenario-aware Semantic Feedback IR}
that structures failures before they become natural-language prompts.

\subsection{Operationalising Prevention within a Bounded Budget}
Full verification is impractical for unannotated LLM output
\cite{leino2010dafny,loughman2024dafnybench}.
The challenge is calibrated prevention: stronger than test-feedback
\cite{chen2021codex}, weaker than deductive proof, and fast enough for sprint
budgets \citep{beck2001xp,li2022agilesofl}.
\textsc{SgDP} addresses this with a conjunctive \emph{acceptance predicate}
and a hard repair budget~$K$ (we use $K=3$ in experiments).


\section{Proposed Approach}
\label{sec:intro:approach}

At a high level, HSP-Agile runs a four-stage loop on each task: synthesise a
candidate from the FSF text, check it against SMT witnesses derived from the
specification, repair with scenario-aware feedback when checks fail, and gate
release with a pattern screen.
Figure~\ref{fig:sgdp-overview} shows the end-to-end flow.
Chapter~\ref{ch:method} defines the general \textsc{SgDP} interface, proves
three correctness results, and walks through \code{classify\_signal} end to
end; Chapter~\ref{ch:results} shows that the same loop transfers to Mini-Z and
Mini-StateMachine via a SpecIR adapter layer.

\section{Contributions}
\label{sec:intro:contributions}

This report makes five concrete contributions
(Figure~\ref{fig:contribution-map}).

\begin{figure}[t]
  \centering
  \tikzinput{diagrams/tikz/contribution_map}
  \caption{Overview of \textsc{SgDP} research contributions.
    The five contributions form an integrated path from pipeline design through
    empirical validation.}
  \label{fig:contribution-map}
\end{figure}

\begin{enumerate}
  \item[\textbf{C1.}] \textbf{Specification-guided Defect Prevention Framework
    (\textsc{SgDP}).}
    A notation-agnostic SpecIR layer with pluggable adapters and a four-stage
    prevention loop; HSP-Agile is the SOFL/FSF instantiation.

  \item[\textbf{C2.}] \textbf{Strict Acceptance Predicate and Semantic Feedback
    IR.}
    Definition of $\mathrm{Accept}(c,t)$ and a nine-field JSON IR that separates
    failure structure from prompt rendering.

  \item[\textbf{C3.}] \textbf{Precedence-Sensitive Hard Benchmark.}
    120 synthetic tasks with overlapping guards, plus real-derived and
    multi-notation subsets.

  \item[\textbf{C4.}] \textbf{Two-Layer Prevention Evaluation Protocol.}
    PDR/FAR measurement via specification-confusion and implementation-screening
    mutants.

  \item[\textbf{C5.}] \textbf{Empirical Evidence and Reproducible Artifacts.}
    90.4\% mean strict conformance (+6.2\,pp over one-shot LLM); full logs and
    scripts released.
\end{enumerate}
