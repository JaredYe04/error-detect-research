\chapter{Introduction}
\label{ch:intro}

Formal specifications buy precision that informal prose cannot match.
That precision becomes a liability when large language models (LLMs) write the
implementation: the model has no native notion of guard order, so it produces
code that is statistically plausible yet formally wrong.
The resulting faults pass most everyday inputs and break exactly where the
specification draws its boundaries.
Test-based repair loops miss them because they sample the input space without
knowing which regions matter.

The question that drives this report is therefore practical:
\emph{can specification-conformance defects in LLM-generated code be prevented
by using the specification itself as the primary evidence source, without full
deductive verification?}
The answer developed below is that such prevention is \emph{feasible} when
specification structure---especially dense guard overlap or conjunctive release
requirements---warrants SMT witnesses and typed repair feedback; it is
\emph{not} a universal substitute for test-feedback repair on sparse-overlap
specifications (Chapters~\ref{ch:results}--\ref{ch:discussion}).
The vehicle is the \textbf{Specification-guided Defect Prevention Framework}
(\textsc{SgDP}), with \textbf{HSP-Agile} as its SOFL/FSF instantiation.


\section{Problem Statement}
\label{sec:intro:problem}

\subsection{Setting}
SOFL \citep{liu1998sofl,liu2004soflbook} provides the Functional Specification
Format (FSF): an ordered list of scenarios, each pairing a guard with a
postcondition under first-match precedence.
Agile-SOFL \citep{li2022qrs_companion} embeds those artefacts in sprint workflows
as design documents, oracles, and acceptance criteria.
LLM-assisted generation accelerates that workflow
\cite{chen2021codex,openai2023gpt4}, but synthesises by pattern completion
rather than by respecting guard order
\cite{dakhel2023copilot,sun2024clover}.
Overlapping guards therefore expose precedence-sensitive faults that standard
test suites miss \citep{zhu1997specification}.

\subsection{What Goes Wrong}
A specification-conformance defect is, in plain terms, a bug that unit tests
rarely catch.
The generated function may look correct on the happy path, yet on a thin slice
of the input space---often where two guards overlap---it returns the output of
the wrong scenario.
Random or distribution-biased tests almost never land in that slice.
A repair loop that only sees failing unit tests never learns about the ordering
constraint.

That observation leads to a release decision.
Given task~$t$ with ordered scenarios $\mathcal{S}_t = (S_1,\ldots,S_n)$ and
synthesis budget~$K$, the question is whether any candidate~$c$ satisfies
\begin{equation}
  \mathrm{Accept}(c, t)
  \;\iff\;
  \mathrm{Conf}(c, t) \geq \tau
  \;\wedge\;
  P^H(c) = 0,
  \label{eq:acceptance}
\end{equation}
where $\mathrm{Conf}(c,t)\in[0,1]$ is the fraction of specification-derived
witnesses on which $c$ matches the oracle, $\tau=1.0$ in this work, and
$P^H(c)=0$ asserts that no high-severity structural pattern remains.
A witness for scenario~$S_i$ must satisfy
$G_i(\mathbf{x}) \wedge \bigwedge_{j<i}\neg G_j(\mathbf{x})$---a conjunction
that standard sampling heuristics cannot reliably reach
\cite{barrett2018smt}.

\subsection{A Concrete Example}
Think of an industrial monitoring alert: negative sensor levels must raise
\texttt{"Error"} before any threshold-based \texttt{"Critical"} path can fire.
The running example \code{classify\_signal} (Listing~\ref{lst:fsf-spec}) is that
contract in miniature---small enough to hold in working memory, rich enough to
expose the ordering conflict that motivates the whole framework.

\inputfsf{listings/classify_signal.fsf}{FSF specification for \code{classify\_signal} (pseudocode notation).}{lst:fsf-spec}

When \code{threshold} is negative, the input
$(\mathit{level}{=}{-}3,\;\mathit{threshold}{=}{-}10)$ satisfies both $G_1$
(\texttt{level < 0}) and $G_2$ (\texttt{level >= threshold}).
FSF ordering mandates \code{"Error"} via~$S_1$.
A typical LLM candidate evaluates the threshold comparison first:

\inputpython{listings/classify_signal_bug.py}{LLM-generated candidate with a scenario-ordering violation.}{lst:llm-incorrect}

The candidate returns \code{"Critical"} instead of \code{"Error"}.
A test suite that draws \code{threshold} from a non-negative distribution never
reaches this region---exactly the latent defect that
Equation~\eqref{eq:acceptance} must prevent.
Figure~\ref{fig:defect-vs-test} makes the sampling contrast concrete: denser
random tests do not help; a specification-aware witness does.

\section{Research Challenges}
\label{sec:intro:challenges}

Three obstacles map directly to the three insights below.

\subsection{Semantic Regions, Not Surface Syntax}
Checking scenario~$S_i$ requires inputs in
$G_i \wedge \bigwedge_{j<i}\neg G_j$---regions random tests miss
\cite{barrett2018smt}.
The check must be notation-agnostic yet preserve first-match partitions
(Chapter~\ref{ch:formalization}).

\subsection{Typed Failures for Repair}
Test-feedback loops lack specification-level structure
\cite{sun2024clover}: the model may patch one counterexample without
fixing guard precedence.
Repair needs a record that names the violated scenario and witness
\emph{before} any natural-language prompt is rendered
(Chapter~\ref{ch:method}).

\subsection{Conjunctive Release under Budget~$K$}
Full verification is impractical for unannotated LLM output
\cite{leino2010dafny,loughman2024dafnybench}.
What is needed is calibrated prevention: stronger than test-feedback
\cite{chen2021codex}, weaker than deductive proof, and fast enough for sprint
budgets \citep{beck2001xp,li2022qrs_companion}---a conjunctive gate under
hard budget~$K$ ($K=3$ in the experiments).


\section{Proposed Approach}
\label{sec:intro:approach}

Why continue past the problem statement?
Because the same specification that exposes the defect can also drive
prevention---if three ideas hold: shared semantic regions, typed repair
feedback, and conjunctive acceptance.
Figure~\ref{fig:sgdp-overview} is the map; adapters and SMT lowering are
necessary engineering behind that map, not separate contributions.
Chapter~\ref{ch:method} walks \code{classify\_signal} end to end;
Chapter~\ref{ch:results} shows the ideas transfer with adapter-level changes.

\section{Contributions}
\label{sec:intro:contributions}

Four insights---and the deployment rule that follows---are what a reviewer
should remember (Figure~\ref{fig:contribution-map}).
Adapters, SMT lowering, the hard benchmark, and PDR/FAR are necessary
engineering; they support the claims but are not advertised as contributions.

\begin{figure}[t]
  \centering
  \tikzinput{diagrams/tikz/contribution_map}
  \caption{\textbf{Key observation:} three Tier-1 insights, not a feature list.
    SpecIR, Semantic Feedback IR, and the conjunctive gate are the memorable
    core; everything else is supporting mechanism.
    \textbf{How to read:} C1 enables shared checking; C2 enables repair; C3
    gates release.}
  \label{fig:contribution-map}
\end{figure}

\begin{enumerate}
  \item[\textbf{C1.}] \textbf{Insight:} prevention should depend on
    \emph{semantic regions} (first-match partitions), not on the surface syntax
    of any one notation.
    SpecIR realises that insight; HSP-Agile is the SOFL/FSF instance.

  \item[\textbf{C2.}] \textbf{Lead insight:} repair needs a \emph{typed scenario
    failure}, not raw pass/fail bits.
    The Semantic Feedback IR names the violated scenario, witness, and expected
    output before any natural-language prompt is rendered
    (+7.7\,pp over test-only feedback in E6).

  \item[\textbf{C3.}] \textbf{Insight:} release must be \emph{conjunctive}:
    every specification-derived witness matches \emph{and} no high-severity
    structural pattern remains.
    $\mathrm{Accept}(c,t)$ is that gate; high mean conformance alone is not
    release.

  \item[\textbf{C4.}] \textbf{Deployment boundary:} practitioners should
    match feedback depth to release requirements---test-feedback (B2) as the
    default for mean conformance and latency; full HSP-Agile (M) when conjunctive
    release or prevention (PDR/FAR) is required, with E6 typed IR as the
    mechanism anchor (Table~\ref{tab:deployment-boundary}).
\end{enumerate}

On the current 120-task hard benchmark, canonical E1 (seed~0) reports B1 at
89.2\%, B2 at 87.7\%, and M at 86.3\%---gaps are not Holm-significant.
E12-full (three seeds) is authoritative for ranking stability: B2 leads every
seed (mean 88.0\% vs.\ M 86.0\%; Friedman $p{=}0.077$, B2 first every seed).\footnote{Prior E1 runs on a superseded
86-task-overlapping corpus reported M at 90.4\%; all canonical numbers use
\texttt{run\_e1\_canonical\_v1} on \texttt{benchmarks/tasks.json}.}
Evaluation protocol details appear in Chapters~\ref{ch:experiments}
and~\ref{ch:results}.


\section{Research Questions}
\label{sec:intro:rq}

Five research questions organise the evaluation.
Results appear in \emph{mechanism-first} order in Chapter~\ref{ch:results}
(why the framework works, then that it works overall); the numbering below
follows the experimental logic of Chapter~\ref{ch:experiments}.

\textbf{RQ1}---does typed Semantic Feedback IR improve repair beyond test-only
feedback? (E6, A3);
\textbf{RQ2}---does the M--baseline gap grow monotonically with guard-overlap
density? (E3, E10);
\textbf{RQ3}---does conjunctive $\mathrm{Accept}$ improve prevention vs.\
mean conformance alone? (E2, A2 vs.\ M);
\textbf{RQ4}---under what conditions should practitioners deploy M vs.\ B2?
(E3, E10, E11, deployment boundary);
\textbf{RQ5}---quality--overhead trade-off (E1 Pareto, A2 operating point).
